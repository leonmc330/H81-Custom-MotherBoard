TVW (Teboview) binary format

Working spec covering both the Gigabyte variant (originally cracked from Z490 / X570 / B550) and the Compal/Lenovo variant (cracked from T480 NM-B501R10). Status badges mark each claim as verified, speculative, or open.

1. Overview #

TVW is the binary boardview format produced by the Teboview viewer (originally a Gigabyte tool, later adopted by other OEMs). All known variants share a core file-level structure but disagree on details — chip-record layout, layer encoding, pad-record stride, and the location of auxiliary tables.

This document covers two variants:

Gigabyte

Verified on motherboard files (Z490 VISION G, X570 GAMING X, B550 AORUS PRO AC) and GPU files (GV-N780OC-3GD 10-layer, GV-R79XTGAMING 12-layer). Two layers TOP/BOTTOM on mobos; multi-layer on GPUs.

Compal/Lenovo

Verified on the Lenovo Thinkpad T480 NM-B501R10 (10 copper layers). Sometimes called the "Compal variant" because Compal is Lenovo's contract manufacturer for the T-series. Structure deviates from Gigabyte in several places — chip records use a different anchor scheme, layer pads are 19-byte stride (not 38-byte), and the master footprint pool starts ~234 KB earlier than the Gigabyte convention would predict.

Both variants are little-endian throughout.

2. Coordinate system #

  • All coordinates are signed 32-bit integers in file units.
  • 1 file unit ≈ 0.32 µm (verified across Gigabyte and Compal/Lenovo; 50 file units ≈ 16 µm matches the empirical pad-matching tolerance).
  • 1 mm ≈ 3,125 file units. 1 mil (0.0254 mm) ≈ 79 file units.
  • Typical motherboard span: ±1,000,000 file units (≈ 320 mm).
  • Coords stored as (Y, X) on disk in Gigabyte trace data, (X, Y) in chip records and master records. Every reader must check which is which per section.
  • Origin (0, 0) is anchored on a mounting hole (MH1 on Gigabyte boards). Real CAD trace endpoints never snap to origin — it's a reference anchor only.

5. Chip enumeration #

In the Compal/Lenovo variant, chip placement uses a single source — Region 3 — while chip layer assignment reads from a 3-source supervised chain:

  1. Region 3 records — the only chip placement source. Carries position, rotation, fp_id, device value, tolerance, part code, and the full inline pin list with display numbers and Pascal pin names. 2,754 placed chips on T480 (1,664 TOP / 1,090 BOTTOM) after filtering pin sub-records, test-point fp_ids (235..240 = OUTLINE_TB_TP1..TP6), and the 01 04 30 2e false-positive record class.
  2. Region 1 supervised layer flag — read-only layer signal for ~296 chips, used during layer determination. Never used for chip placement.
  3. Cap-section trailer byte — read-only layer signal for ~884 chips, used during layer determination. The cap-section's chip-header records themselves are NOT enumerated; only the trailer byte at after_off + 20 is consulted as a layer source for refdes already enumerated via R3.

Earlier drafts of this spec described a "union over 3 chip sources" — that turned out to be unnecessary. Every chip the cap-section and R1 see is also seen by R3; the union didn't gain coverage, only redundancy. The 2,754 R3 chips on T480 are the full board roster.

Filter rule: the parser checks only the single 0x01 byte at +44 as a cheap prefix sentinel and then attempts to walk the 5 Pascal strings. A malformed walk (any Pascal length > 100 or any non-ASCII content) aborts the record, which is sufficient to reject the 01 04 30 2e-class false positives without needing a fixed 4-byte signature match.

Per-pin record (inline)

Variable stride 17 + L bytes, where L is the Pascal length of the pin name:

OffsetTypeField
+0..+3uint32ptr — steps by 8 across pins (U58 first_ptr = 12344)
+4..+74 zeros
+8..+11uint32pin_display — integer shown in BoardViewer.exe's Pin column (e.g. 148 for U58's pin "A148"). The parser uses this verbatim as the pin name.
+12uint8Pascal length L
+13..+12+LL bytesfull pin name (e.g. "A3", "K1", "AA10") — BGA grid coord including row letter; kept for completeness but not used as the display name.
+13+L..+16+L4 zeros

Pin → master link: master_pin_idx = (ptr - first_ptr) // 8. The master record's Section C (per-pin local coords) at this index gives the pin's local (A, B) position.

Why two pin labels? The Pascal string carries the manufacturer's BGA grid notation (e.g. "A148") and the uint32 at +8 carries the equivalent integer position (148). For non-BGA chips (QFN/SOIC) the two collapse to the same value. We expose the integer because that's what BoardViewer.exe shows by default and the ground-truth references (e.g. "U58.148 = USBCOMP") use it.

5.2 Cap-section records #

verified Uses the Gigabyte-style 0x01 + Pascal-dev_name + Pascal-footprint chip-header pattern. The existing _find_chip_headers() in tvw_parser.py detects these. On T480 they cover ~884 chips, mostly small passives (C, P refdes).

The trailer immediately after the chip header (at c['after_off']) starts with an 11-byte Pascal "part code" string like SE000010J0T, SGA00007T00, SH00001AB00 (length prefix 0x0b). On Gigabyte boards this Pascal isn't present and the layer byte sits at after_off + 9; on Compal/Lenovo it's at after_off + 20 (20 bytes past the SE-string start).

Compal/Lenovo cap-section layer byte

At c['after_off'] + 20: 0x02 = TOP, 0x0b = BOTTOM. Verified on 20 chips that overlap with Region 1 supervised data (all 20 agree).

5.3 Region 1 layer-flag table #

verified Compal/Lenovo only. Detected by signature: Pascal-prefixed refdes where the byte sequence b8 0b 00 00 2c 01 00 00 (= constants 0xbb800 and 0x12c00) appears at after-Pascal +12..+19. Layer flag at after-Pascal +24..+27 (int32):

  • 0 = TOP
  • 1 = BOTTOM

Covers 296 unique top-level chips (after filtering pin sub-records). Used as authoritative ground truth — never disagrees with itself across multi-anchor chips. Subset of Region 3.

6. Master footprint records #

6.1 Pool location #

verified Compal/Lenovo: master pool at 0xbd69e3 .. 0xc3fe8b, 240 master records. Each starts with the signature 00 00 00 00 00 00 00 00 01 (8 zeros + 0x01) followed by a Pascal-prefixed footprint name.

Note: earlier investigation missed the first 34 masters in 0xbd69e3..0xbf09c6, leading to wrong fp_id offsets. The correct pool extends from 0xbd69e3, not 0xbf0000.

6.2 Section A–D layout #

Each master record's data section (after the 9-byte signature + Pascal name) decomposes into:

Section A — outline polyline (24-byte records)

+0..+3   sentinel  0xff 0xff 0xff 0xff

Structured as M × subblock + 12-byte footer. Each subblock = 84-byte fixed preamble + variable body. Body:

+0..+3   uint32  leader = 10 + N_shapes (next available enum_id)

6.3 fp_id → master_idx #

verified Rule: master_idx = fp_id - 1. Clean across all 1,452 resolvable chip records. No skips, no orphans.

The chip's stored footprint name is always the BASE name (no _B suffix). The fp_id selects the actual master variant — base or _B. So a chip on BOTTOM that uses a paired footprint will record UM6K33N_UMT6 as its footprint string but its fp_id will point to the master record for UM6K33N_UMT6_B.

1,025 chips map to the base-name master (TOP variant); 427 map to the _B (BOTTOM) variant. 65 masters are dead (never referenced) — library templates retained on disk but unused on this board.

7. Layer pad records #

verified Compal/Lenovo uses 19-byte stride layer pad records for all 10 copper layers AND for the connector / through-hole / BGA-ring runs in the 0x008d8000..0x00914012 gap. Format:

OffsetTypeField
+0..+12 zero bytesrecord sentinel (cheap pre-filter)
+2uint8pad-shape enum (see callout)
+3..+6uint32net_id (< 4000, indexes net-name table)
+7..+10uint32pad_type (> 0, indexes per-region aperture table)
+11..+14int32Y world
+15..+18int32X world

~180,000 records total across 10 layer regions + the gap region. 40,031 unique (Y, X) positions (up from 28,856 when we required a 3-zero prefix). Pads appearing in multiple regions = through-vias.

Pad-shape enum (+2 byte):
  • 0x00 — standard SMD surface pad (150,338 records)
  • 0x01 — SMD pad, alternate footprint variant (2,742 records)
  • 0x02 — through-hole / via-attached connector pin (10,727 records)
  • 0x05 — BGA-style pad with annular ring (9,583 records)
  • 0x08 — press-fit / mechanical-mount pin, e.g. dock / USB-C ground tab (4,197 records)

Earlier drafts of this spec treated +0..+2 as a 3-zero sentinel. That hid 11,175 unique pad coords concentrated in the 0x008d8000..0x00914012 region (between GND1 end and BOTTOM trace start) — primarily connector through-holes and mounting tabs. Without those, all class-17 connectors (JKBL1, JTAG2, JDDR1, ...) returned empty pin → net mappings.

Common pitfall: The Gigabyte 38-byte pad-record scanner, applied naïvely to Compal/Lenovo, reads two adjacent 19-byte records as one 38-byte record. The "first half" looks like garbage (it's another pad's data), and the count comes out at exactly half. Confirm format by checking the 2-byte zero prefix + plausible net_id + non-zero pad_type.

9. Vias and drills #

verified Compal/Lenovo has no dedicated via record type. Vias are inferred from layer-pad XY coincidence: a pad position appearing in ≥ 2 layer regions = a via at that XY connecting those layers.

Pad coincidence patterns:

  • 11,153 unique positions in only one region (pure SMD pad)
  • 6,714 unique positions in all 8 inner copper layers (buried/blind inner-only via stack)
  • 164 unique positions in all 10 layers (true through-hole — SO-DIMM, ATX-like connectors, mounting holes)
  • 469 in 8 inner + TOP only (top-side blind via)
  • 73 in 8 inner + BOTTOM only (bottom-side blind via)

GND-stitching drill table (38-byte stride)

The exception. At 0x985d9 .. 0xa53bb, 1,387 records:

+2..+5    uint32  net_id    (always 45 = GND for this board)

10. Net-name table #

verified Compal/Lenovo location: 0xb384fe.

  • Header (8 bytes): d6 0a 00 00 d6 0a 00 00 = uint32 count (= 2,774) repeated twice.
  • Strings at 0xb38506: sequential Pascal-prefixed net names. net_id is 0-based — nets[0] is the first Pascal name in the table. No leading placeholder entry.

Verified net_ids from BoardViewer.exe ground truth: GND = 45, CPU_PWRGD = 561, VCCCPUCORE = 968, TBT_LSTX = 1005, TBT_LSRX = 1006, PECI = 1330, USBCOMP = 1446. After implementing 0-based lookup, all 9 ground-truth pin → net pairs supplied by the issue reporter pass (see section 17).

Indexing pitfall. An early draft of the parser used nets = [""] + ... (1-based with a leading empty placeholder). That shifted every reported net by one and produced U58.148 = "USBP2+" instead of "USBCOMP". The 0-based table indexing is confirmed by checking 9 unrelated chip+pin pairs — there is no leading entry, the first Pascal in the table really is net_id = 0.

11. Pin position and net mapping #

11.1 Canonical pin-position transform

verified The transform is heuristic-free — no 16-way rotation search. Residuals are exactly 0 on every pin whose pad exists in the layer-pad records.

def chip_pin_to_world(chip, A_local, B_local, layer):
    """
    chip['f2'], chip['f3']: the (Y_min, X_min) corner of the 16-byte
                            bbox field at the chip R3 record's
                            after-Pascal +8..+23.
    (A_local, B_local):      the two int32s from a 19-byte master
                            Section C record.
    chip['chip_class']:      uint32 at R3 +32..+35. Connectors
                            (class=17) require an additional negate
                            step (see below); all other classes use
                            the canonical rule.
    """
    rot = chip['rot']
    if   rot ==   0: dy, dx = -A_local, -B_local
    elif rot ==  90: dy, dx = -B_local,  A_local
    elif rot == 180: dy, dx =  A_local,  B_local
    elif rot == 270: dy, dx =  B_local, -A_local

    # Connector class — pin frame matches canonical only for the two
    # "natural" placements (rot ∈ {0, 180} on BOTTOM). Other (rot, layer)
    # combinations require an extra negation BEFORE the BOTTOM flip.
    if chip['chip_class'] == 17:
        if not (rot in (0, 180) and layer == 'BOTTOM'):
            dy, dx = -dy, -dx

    if layer == 'BOTTOM':
        dx = -dx        # BOTTOM-side chips mirror world-X delta
    return (chip['f2'] + dy, chip['f3'] + dx)

Verification: 216 chips across all 4 rotations × both layers × diverse component types (BGA, QFN, MOSFET, connector, diode) — every pin whose pad exists matches with 0 residual. Connector-specific verification: JKBL1 (38/38 pins), JTAG2 (8/8 pins), plus pin-1 spot checks on JLAN1, JDOCK1, JHDMI1, JUSBC1, J41 — all match with 0 residual.

Why class-17 needs the extra negate (heuristic explanation)

For ICs, the chip's bounding-box anchor (f2, f3) happens to align with the "logical pin 1 corner" of the IC master at every rotation. For connectors, the anchor aligns with the OPPOSITE corner in the four non-canonical placements (TOP-side, or 90°/270° on BOTTOM). The single-line negate restores the consistent anchor → pin-1 alignment without needing a 6-row (rot, layer)-keyed lookup table.

About the 16-byte bbox field: the field at chip R3 +8..+23 contains 4 int32s (f0, f1, f2, f3). The semantics differ wildly across component types — for BGAs it's a constant 700×700 box; for QFN/MOSFET it approximates the chip body; for connectors it's a long thin rectangle. The values look inconsistent because they probably encode different things per chip-type, but (f2, f3) consistently works as the canonical pin anchor. Treat (f0, f1) as the TR-corner of the chip-body bbox (vs (Y, X) as the BL-corner) — useful for placement/bbox rendering, not for pin transforms.

11.2 Pin → net via spatial matching

verified There is no direct pin → net lookup table in Compal/Lenovo TVW. The chip's inline pin list (per-pin records inside the R3 record) only carries pin numbers and Pascal names — never a net_id. Net assignment is performed entirely by matching the predicted pin world coords against the file's 19-byte layer-pad records.

Algorithm

  1. Walk every 19-byte layer-pad record in the file (see section 7) and build a deduped spatial index {(world_Y, world_X) -> net_id}. T480: ~180,000 pad records collapse to 40,031 unique (Y, X) keys after deduplication across the 10 copper layers plus the gap region.
  2. For each chip pin, compute its predicted world coord using the transform from §11.1 (with the class-17 negate if applicable).
  3. Exact-integer lookup in the spatial index. Hit → net_name = nets[net_id] (0-based); miss → pin is no-connect (NC) and is not added to model.signals.

Exact-coord matching works because chip (Y, X), (f2, f3), and master-local (A, B) are all integer file units, and rotations are exact multiples of 90°. The transform output is deterministic with zero residual — no tolerance or nearest-neighbour search is needed, no quantisation rounding to coarse mils. The whole pipeline is bit-exact.

Reference implementation

def build_pad_index(data):
    """Single linear pass over the file. Two-zero prefix is the
    cheap pre-filter; the +2 byte is a pad-shape enum, NOT a record
    sentinel — see §7 callout."""
    index = {}
    i = 0
    n = len(data) - 19
    while i < n:
        # Only 2 zero bytes required at +0..+1. The +2 byte is a
        # pad-shape enum (0=plain, 1=variant, 2=through-hole,
        # 5=BGA ring, 8=press-fit). Filtering on +2==0 hid 11k
        # connector / via / through-hole pads.
        if data[i] != 0 or data[i+1] != 0:
            i += 1
            continue
        net_id   = int.from_bytes(data[i+3 :i+7 ], "little")
        pad_type = int.from_bytes(data[i+7 :i+11], "little")
        Y        = int.from_bytes(data[i+11:i+15], "little", signed=True)
        X        = int.from_bytes(data[i+15:i+19], "little", signed=True)
        # pad_type==0 is the most common false positive (zero-padded
        # alignment slack between regions). Real pad_types start at 1.
        if (net_id  < 4000
            and 0 < pad_type < 100000
            and abs(Y) < 2_000_000
            and abs(X) < 2_000_000):
            index.setdefault((Y, X), net_id)
        # Byte-aligned advance — the gap-region records don't sit on
        # a stride-19 grid relative to file position 0. The dedup via
        # setdefault handles overlapping false-matches cleanly.
        i += 1
    return index

def chip_pin_to_net(chip, A, B, layer, nets, pad_index):
    world_yx = chip_pin_to_world(chip, A, B, layer)
    net_id = pad_index.get(world_yx)
    if net_id is None or net_id >= len(nets):
        return None                       # no-connect
    return nets[net_id]                   # 0-based

Bounds (the 4000 / 100000 / 2,000,000 thresholds)

  • net_id < 4000 — T480 has 2,774 nets; the threshold is loose enough to admit any plausible motherboard but tight enough to reject random 32-bit values inside other record types that happen to share the 2-zero prefix.
  • 0 < pad_type < 100000 — aperture indexes; in practice all observed values are 1..~280 for layer aperture entries (master pad-shape references can go higher but stay well under 100k). pad_type == 0 is rejected to filter out zero-padded alignment slack between record runs.
  • |Y|, |X| < 2,000,000 file units — ≈ 640 mm board span limit. T480 board envelope is roughly ±900,000 fu.

Coverage and NC rate on T480

  • 2,754 chips, sum of pin counts (from R3 inline pin lists) ≈ 12,000+ after the master-pin off-by-one fix
  • Spatial match hits: 10,203 — produces model.signals entries (was 7,149 before the §7 prefix relaxation, §5.1 chip_class capture, §11.1 class-17 transform, and the master-pin off-by-one fix all landed)
  • 2,717 unique nets populated (out of 2,774 in the name table). Remaining 57 are typically NC test points or board-internal scratch nets.
  • 19/19 ground-truth pin → net pairs pass on the current parser (was 9/9 before the connector + chip_class work).
What about pads that map to multiple net_ids? Pathological cases would occur if two different nets had pads at the exact same world coord. We don't observe this on T480 (the setdefault in build_pad_index never overwrites a populated slot in practice). If a future file did trip this, the index would return whichever net_id arrived first — acceptable for an MVP, but a future refinement would layer-scope the index by reading pad-record layer membership from the per-layer aperture-table runs.

12. Layer flag — three-source chain #

verified No single byte inside the chip R3 record encodes layer (best single-bit predictor across 290+ supervised chips reaches only 81.9% accuracy vs 76.4% baseline). Layer must come from:

  1. Region 1 supervised flag (int32 at after-Pascal +24..+27): 0 = TOP, 1 = BOTTOM. Covers 296 chips.
  2. Cap-section trailer byte at c['after_off'] + 20: 0x02 = TOP, 0x0b = BOTTOM. Covers 884 chips.
  3. Master _B suffix: the chip's fp_id resolves to a master record; if that master's name ends in _B the chip is BOTTOM, else TOP. Covers chips with paired footprints (~427 chips).

For chips covered by multiple sources, all sources agree. Coverage union: ~2,200+ chips (essentially all real chips on T480). Chips covered by NONE of the three (e.g., fiducials, test points) default to TOP — acceptable since they have no electrical significance.

13. Filter rules #

verified Two filter rules essential to avoid spurious matches:

Chip-record sentinel + structural-walk filter

Valid R3 chip records have a single 0x01 sentinel byte at chip-record +48 (i.e. after-Pascal +44). A "second class" of records exhibits 01 04 30 2e at the same offset and has nonsense pin counts (e.g. 16,777,216) — not real chips. Rather than requiring a fixed 4-byte signature match, the parser uses:

  1. data[+44] == 0x01 — cheap byte sentinel.
  2. Attempt to walk the 5 packed Pascal strings (device value, tolerance, tolerance dup, footprint name, part code) — see §5.1. Any Pascal length > 100 or any non-ASCII content aborts.
  3. fp_id − 1 must index a real master record in the pool.
  4. fp_id not in {235..240} (test-point list).
  5. |Y|, |X| < 2,000,000 file units.

This rejects the 01 04 30 2e class cleanly via the Pascal-walk step — those records' bytes immediately following +44 don't parse as valid Pascal strings.

Pin sub-record filter

The R3 chip scanner picks up some pin sub-records of larger chips as if they were standalone chips. Earlier filter drafts used a broad prefix allow-list, but most of the rejected prefixes turned out to be real chip refdes on T480 (PQ = MOSFETs, PD = diodes, PL = inductors, etc.). The shipped filter is much narrower — a single regex matching only:

^(H[0-9]+ | FL[0-9]+ | FLJ[0-9]+)$

The structural pre-filter (single-byte sentinel + valid Pascal walk + valid fp_id + valid coord range) does most of the work; this regex catches the few pin sub-records that nonetheless survive the structural checks. Verified clean on T480 — no real chips dropped, no pin sub-records leaked through.

Test-point fp_id filter

fp_ids 235..240 correspond to OUTLINE_TB_TP1..TP6 — test-point list entries with no chip body, just outline placeholders. ~1,621 such records exist; they have no electrical meaning and should be filtered.

14. Variant detection #

To select the right parser path at runtime, detect which variant the file is. Several discriminators are available:

Compal/Lenovo signatures

  • Pascal-prefixed SE...T, SGA...T, or SH...T 11-char strings present in cap-section trailers. Look for the byte sequence 0x0b 'S' 'E' / 'G' / 'H' following the chip-header marker.
  • The chip-record magic flag 01 01 2a 00 appears at chip R3 +44..+47.
  • Region 1 layer-flag records: scan for the byte sequence b8 0b 00 00 2c 01 00 00 at any after-Pascal +12..+19 location. If > 100 such records exist, it's a Compal/Lenovo variant.
  • Master pool start: walk back from end of file looking for 00 00 00 00 00 00 00 00 01 [Pascal name]. If first master is at ~0xbd6000 rather than near the file end (Gigabyte convention is > 80% through file), it's Compal/Lenovo.

Gigabyte signatures

  • The Custom_35 and Custom_17 Pascal markers appear with payload-size gaps > 1 MB (matching the dedicated TOP/BOTTOM trace regions).
  • No Region 1 layer-flag records found.
  • Cap-section chip trailer byte +9 is bimodal 0x02/non-0x02 (Gigabyte layer flag).
  • Master pool sits at > 80% through file.

Implemented detection (see tvw_parser._detect_variant): data.count(b"\xb8\x0b\x00\x00\x2c\x01\x00\x00") > 100 → Compal/Lenovo. Otherwise → Gigabyte (the historical default).

The default-to-Gigabyte fall-through is intentional: producing a (potentially imperfect) Gigabyte parse is preferable to silently returning an empty model on a misclassified file. Compal-variant detection is intentionally conservative (high threshold = 100) to avoid false positives on Gigabyte GPU files which can contain unusual byte patterns in their wider trace regions.
